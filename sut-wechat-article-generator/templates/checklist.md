# 白话化检查清单

## 使用说明

本文档用于检查文章是否符合"白话为主、降低阅读门槛"的要求。

在完成文章后，逐项检查，确保内容通俗易懂、易于理解。

---

## 第一部分：标题检查

### 1.1 长度检查
- [ ] 标题长度在 15-25 字之间
- [ ] 如果超过 25 字，考虑简化
- [ ] 如果少于 15 字，考虑补充信息

### 1.2 吸引力检查
- [ ] 包含关键词（主题核心）
- [ ] 有数字或疑问词
- [ ] 利益点明确（告诉读者能获得什么）
- [ ] 避免标题党式夸大

### 1.3 白话化检查
- [ ] 没有使用生僻的专业术语
- [ ] 没有使用英文缩写（除非是广为人知的）
- [ ] 语言口语化，易于理解

**示例：**
- ❌ "基于 Transformer 架构的大语言模型在代码生成任务中的应用研究"
- ✅ "5分钟读懂 Transformer，AI 代码生成背后的秘密"

---

## 第二部分：开头钩子检查

### 2.1 注意力检查
- [ ] 能否在3秒内抓住注意力
- [ ] 第一句话是否有吸引力
- [ ] 避免了"今天我们来聊聊..."等陈词滥调

### 2.2 相关性检查
- [ ] 与目标读者的痛点/需求相关
- [ ] 建立了读者与主题的连接
- [ ] 让读者觉得"这篇文章与我有关"

### 2.3 好奇心检查
- [ ] 制造了悬念或好奇心
- [ ] 让读者想要继续阅读
- [ ] 避免在开头就透露全部内容

### 2.4 白话化检查
- [ ] 使用了生活化的场景或类比
- [ ] 避免了学术化表达
- [ ] 语言简洁生动

**示例：**
- ❌ "本文旨在探讨大语言模型在代码生成领域的应用现状"
- ✅ "每天写代码写到手软？AI 助手能让你的效率提升3倍"

---

## 第三部分：内容白话化检查

### 3.1 技术术语处理

**检查规则：**
- [ ] 专业术语在第一次出现时都有解释
- [ ] 使用了生活化类比帮助理解
- [ ] 类比后补充了准确定义
- [ ] 避免连续使用多个术语

**术语处理三步法：**
1. 识别难懂术语
2. 寻找生活类比
3. 补充准确定义

**示例检查：**
```
❌ 未处理：该系统采用RESTful API架构，实现前后端分离。

✅ 已处理：这个系统像一个外卖平台：
前端是顾客下单，后端是餐厅制作，
API就是骑手负责传递信息。

> 📖 **准确定义**：RESTful API 是一种软件架构风格，
> 实现前后端分离和状态管理。
```

---

### 3.2 句式优化检查

**避免学术化表达：**
- [ ] 没有"本文旨在探讨..."
- [ ] 没有"值得注意的是..."
- [ ] 没有"综上所述..."
- [ ] 没有"毋庸置疑..."

**使用口语化连接词：**
- [ ] "换句话说" → "说白了"
- [ ] "具体来说" → "举个例子"
- [ ] "然而" → "不过"
- [ ] "因此" → "所以"

**控制句子长度：**
- [ ] 理想长度：15-25字
- [ ] 最长不超过：35字
- [ ] 超长句子已拆分为多个短句

**使用主动语态：**
- [ ] "数据被系统处理" → "系统处理数据"
- [ ] "功能被用户调用" → "用户调用功能"

**示例：**
```
❌ 学术化：
本文旨在探讨 Transformer 模型在自然语言处理领域的应用。
值得注意的是，该模型采用了自注意力机制。
综上所述，Transformer 是一个重要的技术突破。

✅ 口语化：
今天我们来聊聊 Transformer 是怎么改变 NLP 的。
这个模型有个核心设计叫"自注意力机制"。
说白了，Transformer 标志着 NLP 领域的重要转折。
```

---

### 3.3 类比使用检查

**使用生活化类比时检查：**
- [ ] 读者熟悉这个类比场景
- [ ] 类比与技术概念相似度高
- [ ] 避免过度简化
- [ ] 类比后补充了准确定义

**没有使用类比时检查：**
- [ ] 是否有难以理解的概念？
- [ ] 这些概念是否适合用类比解释？
- [ ] 能否找到合适的生活化类比？

**示例检查：**
```
✅ 好的类比：
API 就像餐厅服务员，连接顾客和厨房。

❌ 不好的类比：
API 就像操作系统中的系统调用接口。
（问题：用"系统调用接口"解释"API"，读者还是不懂）
```

---

## 第四部分：结构可读性检查

### 4.1 段落控制
- [ ] 每段不超过4行
- [ ] 段落之间空一行
- [ ] 重要内容单独成段
- [ ] 没有超长的段落块

**示例：**
```
❌ 不好：
Transformer 是一种基于自注意力机制的深度学习架构，
它通过编码器-解码器结构处理序列数据，在机器翻译、
文本生成等任务中表现出色，成为了 NLP 领域的核心技术。

✅ 好：
Transformer 是一种深度学习架构。

它通过自注意力机制处理序列数据，
在机器翻译、文本生成等任务中表现出色。

如今，Transformer 已成为 NLP 领域的核心技术。
```

---

### 4.2 视觉节奏
- [ ] 每300字有视觉元素（列表、引用、代码、图片）
- [ ] 避免连续3段纯文字
- [ ] 使用加粗强调关键词
- [ ] 视觉元素与内容匹配

**视觉元素清单：**
- [ ] 列表（- 或 1.）
- [ ] 引用（> 💡）
- [ ] 代码块（```）
- [ ] 配图建议
- [ ] 表格
- [ ] 流程图

---

### 4.3 信息密度
- [ ] 先结论后细节
- [ ] 先通俗后专业
- [ ] 先整体后局部
- [ ] 避免信息过载

**示例：**
```
✅ 好的顺序：
1. 结论：Transformer 让机器翻译质量大幅提升
2. 通俗解释：它像一个翻译团队，每个词都协作
3. 专业说明：基于自注意力机制的架构
4. 细节展开：具体的技术实现

❌ 不好的顺序：
1. 技术细节：Transformer 的数学公式...
2. 架构说明：编码器-解码器结构...
3. 应用场景：机器翻译等...
4. 结论：所以它很重要
```

---

## 第五部分：视觉元素检查

### 5.1 配图建议
- [ ] 开头有封面图建议
- [ ] 功能章节有示意图建议
- [ ] 原理章节有流程图建议
- [ ] 配图提示词具体明确

### 5.2 代码块
- [ ] 代码块有语言标签
- [ ] 代码块前有说明文字
- [ ] 代码格式正确
- [ ] 关键部分有注释

### 5.3 引用
- [ ] 引用样式使用恰当（💡📖⚠️）
- [ ] 引用内容准确
- [ ] 引用前后有过渡
- [ ] 避免过度使用

### 5.4 列表和表格
- [ ] 列表项目不超过7项（如超过，分组）
- [ ] 表格格式正确
- [ ] 表格有标题说明
- [ ] 数据准确无误

---

## 第六部分：互动结尾检查

### 6.1 总结部分
- [ ] 用3-4句话总结核心要点
- [ ] 总结简洁有力
- [ ] 没有引入新内容
- [ ] 与开头呼应

### 6.2 互动设计
- [ ] 选择了合适的互动模板
- [ ] 互动问题具体明确
- [ ] 降低了互动门槛
- [ ] 语气真诚不勉强

### 6.3 行动号召
- [ ] 如果有价值，有明确的行动号召
- [ ] 行动号召清晰（在看/转发/关注）
- [ ] 不要过度要求
- [ ] 尊重读者选择

**互动检查示例：**
```
✅ 好的互动：
你在编程时遇到过哪些效率问题？
AI 工具帮你解决了吗？
欢迎在评论区分享你的经验 👇

❌ 不好的互动：
请务必在评论区留言，点赞收藏转发关注，
否则我会很难过，也不会再更新了。
（问题：道德绑架，给读者压力）
```

---

## 第七部分：整体质量检查

### 7.1 准确性检查
- [ ] 技术信息准确无误
- [ ] 数据和事实有依据
- [ ] 链接和引用正确
- [ ] 没有夸大或误导

### 7.2 完整性检查
- [ ] 回答了读者的核心问题
- [ ] 信息完整不缺失
- [ ] 逻辑连贯无断层
- [ ] 没有明显的遗漏

### 7.3 风格一致性
- [ ] 语气风格统一
- [ ] 称谓保持一致（你/您）
- [ ] 格式规范统一
- [ ] 视觉元素风格协调

### 7.4 目标读者匹配
- [ ] 内容适合目标读者（技术从业者/科技爱好者）
- [ ] 深度适中（不太浅也不太深）
- [ ] 术语有解释
- [ ] 降低了阅读门槛

---

## 快速检查清单（5分钟版本）

如果时间紧张，可以使用这个快速版本：

### ✅ 必须检查项

**标题（30秒）：**
- [ ] 15-25字
- [ ] 有吸引力
- [ ] 无生僻术语

**开头（30秒）：**
- [ ] 3秒抓住注意力
- [ ] 有生活化场景

**内容（2分钟）：**
- [ ] 术语都有解释或类比
- [ ] 段落不超过4行
- [ ] 每300字有视觉元素
- [ ] 句式口语化

**结尾（1分钟）：**
- [ ] 有互动设计
- [ ] 有行动号召（可选）

**整体（1分钟）：**
- [ ] 通读一遍，语感流畅
- [ ] 没有明显的障碍点
- [ ] 符合"白话为主"原则

## 常见问题自查

### Q1：读者说"看不懂"

**检查项：**
- [ ] 是否有未解释的术语？
- [ ] 句子是否太长？
- [ ] 是否缺少类比？
- [ ] 信息密度是否过高？

**解决方法：**
1. 给术语加上类比和定义
2. 拆分长句
3. 增加视觉元素
4. 降低信息密度

---

### Q2：读者说"太啰嗦"

**检查项：**
- [ ] 是否有重复内容？
- [ ] 段落是否过长？
- [ ] 是否有冗余的解释？

**解决方法：**
1. 删除重复内容
2. 拆分长段落
3. 合并相似信息
4. 删除不必要的铺垫

---

### Q3：读者说"太学术"

**检查项：**
- [ ] 是否用了学术化句式？
- [ ] 是否直接用术语而不解释？
- [ ] 是否像写论文？

**解决方法：**
1. 改为口语化表达
2. 术语加类比解释
3. 增加生活化案例
4. 使用"你"而不是"本文"

---

### Q4：读者说"没收获"

**检查项：**
- [ ] 是否提供了实用价值？
- [ ] 是否有独特观点？
- [ ] 是否有可操作的建议？

**解决方法：**
1. 明确文章的价值点
2. 增加实用技巧
3. 提供行动指南
4. 分享个人经验

---

## 检查流程建议

### 初稿完成后（第一次检查）
1. 使用"快速检查清单"（5分钟版本）
2. 重点检查明显的白话化问题
3. 修改明显的学术化表达

### 修改后（第二次检查）
1. 使用完整版检查清单
2. 逐项检查，不留死角
3. 标记需要进一步优化的部分

### 发布前（最终检查）
1. 通读全文，检查语感
2. 重点检查标题和开头
3. 检查互动结尾设计
4. 确认无错别字和格式问题
